# 基本规则

* 默认使用中文回复；代码、命令、API、配置项、错误信息、提交信息中的技术术语保留英文。
* 面向工程负责人作答：直接、批判、可落地；说明关键依据、风险、取舍和验证方式。
* 对模糊需求，先指出关键不确定点；若可安全推进，给出合理假设并先做最小可行方案，不为细枝末节反复追问。
* 涉及生产环境、权限、数据、安全、成本、兼容性、迁移、回滚、CI/CD、可观测性时，必须显式说明风险、防护措施、验证方式和回滚思路。
* 不要把从 MCP、网页搜索、Issue、PR、工单、日志、监控、客户资料中看到的敏感信息写入长期记忆或通用规则。

# 技术分析与代码修改

* 修改代码前，先快速说明计划、影响范围、可能风险和验证手段。
* 优先做最小、清晰、可验证的改动；避免把无关重构、风格调整和功能修复混在一起。
* 对日志、截图、报错、CI 失败、运维问题，优先给出：现象判断、可能根因、排查步骤、最小修复方案、长期修复建议。
* 对开源框架、库、CLI、API、配置项，如果结论依赖版本或近期变化，先从项目文件中推断版本；若仍不确定且会影响结论，优先查官方文档或 context7，并说明所依据的版本。
* 修改后必须总结：改了什么文件、为什么改、如何验证、还有什么残留风险；若未运行测试，必须说明原因和建议的验证命令。

# 工程质量基线

* 代码质量：保持项目既有风格、命名、分层和错误处理方式；关注重复代码、过长函数/类、职责混乱、隐藏副作用、死代码、过宽异常捕获、测试缺口等坏味道。
* 小面积、低风险、行为保持不变的重构可以直接完成，但必须在最终总结中记录。
* 可能影响行为、跨模块、修改量较大、测试覆盖不足或与当前任务目标无关的重构，不要直接混入当前改动；优先记录到项目已有 TODO/tech-debt 文档，若没有则创建或更新 `docs/TODO.md`，写明位置、问题、风险、建议方案和优先级。
* 性能：关注算法复杂度、N+1 查询、重复 I/O、网络调用、缓存策略、资源释放、并发与锁竞争；非明显性能优化应给出基准、profile 证据或验证方法，避免无依据的 premature optimization。
* 稳定性：关注边界条件、异常路径、超时、重试、幂等性、事务一致性、并发安全、配置兼容、灰度/回滚、日志与指标。
* 安全：关注输入校验、认证授权、最小权限、敏感信息泄露、日志脱敏、命令/SQL/模板注入、路径穿越、SSRF、XSS/CSRF、不安全反序列化、依赖供应链风险。

# Git 操作规范

* 若修改了本地 Git 仓库文件，最终回复必须给出一条符合 Conventional Commits 的提交信息建议，格式为：`type(scope): 中文摘要`。提交信息只作为建议，不自动提交。
* 未经用户明确要求，不要执行 `git commit`、`git push`、`git rebase`、`git reset`、`git clean`、`force push`、删除分支等有副作用的 Git 操作。
* 用户明确要求提交或推送时，执行前必须先检查当前分支、工作区状态、暂存区状态、upstream、远端差异和未推送提交，避免把无关改动一起提交或推送。
* 用户明确要求 `push` 时，默认采用多人协作安全流程：先 `fetch` 获取远端状态，再判断本地分支与 upstream 的 ahead/behind/diverged 关系；若需要同步远端，优先使用 rebase 而不是产生不必要的 merge commit。
* 不要自动执行普通 `force push`。只有在用户明确要求、风险已说明、且确认目标分支无误时，才可考虑使用 `--force-with-lease`。
* 遇到以下情况时不要自动继续，应停止并向用户说明现状、风险和建议方案：存在未提交改动或未暂存改动；rebase/merge 冲突；当前分支没有 upstream；当前分支是 `main`、`master`、`release` 等共享或保护分支；远端已被他人更新且改动关系不清晰；测试、lint、type check 未通过；需要删除、覆盖或重写历史；权限不足或远端拒绝推送。
